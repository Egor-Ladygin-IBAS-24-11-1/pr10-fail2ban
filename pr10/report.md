## 3. Механизм блокировки

fail2ban физически блокирует IP через **iptables** — добавляет правило в цепочку `f2b-sshd`, которое DROP-ает пакеты с забаненного адреса.

## 4. Усиленная защита

Новые параметры: `maxretry=2, findtime=300, bantime=1800, bantime.increment=true, bantime.factor=2`

- Разница во времени срабатывания: бан наступил **в 1.5 раза быстрее** (после 2 попыток вместо 3)
- `bantime.increment` — прогрессивное увеличение времени бана при повторных нарушениях (первый бан 30 мин, следующий ×2 = 1 час, затем 2 часа и т.д.)

## 5. Suricata

Своё правило (sid 1000001):
alert tcp any any -> $HOME_NET 22 (msg:"SSH brute-force attempt - multiple SYN"; flow:to_server; flags:S; threshold:type both, track by_src, count 5, seconds 60; sid:1000001; rev:1;)
**Сработавший алерт из fast.log:**  
![fast.log](screens/12_suricata_alert.png)

Дополнительные поля в `eve.json` по сравнению с `fast.log`:
- `timestamp` — точное время события
- `src_ip` / `dest_ip` / `src_port` / `dest_port` — адреса и порты
- `alert.signature_id` — SID правила
- `alert.severity` — уровень серьёзности
- `flow_id` — идентификатор потока

## 6. Сравнение fail2ban vs Suricata

| Вопрос | fail2ban | Suricata |
|--------|----------|----------|
| На каком этапе заметил атаку | После 3 неудачных попыток авторизации (анализ логов SSH) | После 5 SYN-пакетов на порт 22 за 60 сек (анализ сетевого трафика) |
| Что увидел в логе | `Ban <IP>` — факт блокировки | `alert` с msg "SSH brute-force attempt" |
| Заблокировал ли атаку автоматически | **Да** — добавил правило iptables DROP | **Нет** — только алерт (режим IDS, не IPS) |
| Какую информацию дал | IP атакующего, имя тюрьмы, кол-во попыток | IP, порт, SID правила, severity, flow_id, протокол |

## 7. Связь с нормативкой ФСТЭК

| Группа мер | Как реализована |
|------------|-----------------|
| **АУД** (аудит событий) | journalctl fail2ban + eve.json Suricata фиксируют все попытки и алерты |
| **УПД** (управление доступом) | fail2ban блокирует доступ по IP через iptables после N неудач |
| **ЗИС** (защита от вредоносного ПО / атак) | Suricata обнаруживает аномальную сетевую активность (брутфорс по SYN) |
| **РЕГ** (регистрация событий) | fast.log + eve.json — централизованный журнал инцидентов |

## Выводы

1. **fail2ban** — это HIDS/IPS уровня хоста: реагирует на события в логах приложений, **активно блокирует** атакующего через iptables. Эффективен против медленных брутфорсов.
2. **Suricata** — это классический NIDS сетевого уровня: анализирует трафик **до** того, как он дойдёт до приложения. По умолчанию только **алертит**, но в режиме IPS (inline) может и блокировать.
3. Прогрессивный `bantime.increment` критически важен — он экспоненциально увеличивает стоимость повторной атаки для злоумышленника.
4. Комбинация обоих инструментов даёт эшелонированную защиту: Suricata видит атаку на сетевом уровне (даже если SSH ещё не успел залогировать), а fail2ban её физически пресекает.
5. Неожиданно: Suricata-правило на SYN-пакетах сработало **раньше**, чем fail2ban забанил IP — потому что сетевый уровень видит трафик до приложения.

## Контрольные вопросы

1. **IDS vs IPS**: IDS только обнаруживает и алертит, IPS — ещё и блокирует. fail2ban — это именно IPS (блокирует через iptables), хотя формально классифицируется как HIDS.
2. **localhost**: в `ignoreip` по умолчанию стоит `127.0.0.1/8`. Чтобы тестировать на localhost, нужно либо убрать его из ignoreip, либо добавить свой IP в ignoreip и атаковать с другого хоста.
3. **maxretry=1000, findtime=1**: защита **не работает** — окно в 1 секунду нереалистично, за него невозможно набрать 1000 попыток. Злоумышленник спокойно подберёт пароль.
4. **Сигнатуры vs аномалии**: сигнатуры ищут известные шаблоны (как `failregex` в fail2ban), аномалии — отклонения от нормы. Моё правило Suricata — **пороговое сигнатурное** (count 5 за 60 сек).
5. **"Suricata стоит = брутфорс невозможен"**: **не прав**. Suricata по умолчанию в режиме alert — она только уведомляет, но не блокирует. Без IPS-режима (inline) или связки с fail2ban/iptables атакующий продолжит попытки.
6. **threshold**: `type both` — срабатывает на каждое 5-е событие И после этого каждые 60 секунд; `track by_src` — считает отдельно по каждому IP-источнику; `count 5, seconds 60` — порог 5 событий за 60 секунд.
7. **bantime.increment**: без него атакующий знает, что бан всегда одинаковый (например, 10 минут) и может просто ждать. Прогрессивный бан делает каждую следующую попытку экспоненциально дороже по времени — это ломает экономику брутфорса.
